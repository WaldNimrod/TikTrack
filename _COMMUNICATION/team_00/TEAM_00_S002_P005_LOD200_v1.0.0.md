#### R2.1 — Gate Sequence Integrity (Option C — PRIMARY)

**What it detects:** A gate is marked `completed` in `pipeline_state.json` but the expected output artifact is missing from disk. Catches: manual state file edits, `pipeline_run.sh pass` abuse, bypassed gate work.

**Required:** Add `GATE_ARTIFACT_PATHS` map to `pipeline.py`:

```python
GATE_ARTIFACT_PATHS: dict[str, str | None] = {
    # key = gate_id, value = path pattern relative to REPO_ROOT
    # None = no artifact required for this gate (virtual/waiting gates)
    "GATE_0":    "_COMMUNICATION/agents_os/prompts/GATE_0_prompt.md",
    "GATE_1":    None,  # LLD400 path varies per WP — skip for now
    "GATE_2":    None,  # human approval — artifact is in _Architects_Decisions, path varies
    "WAITING_GATE2_APPROVAL": None,
    "G3_PLAN":   "_COMMUNICATION/agents_os/prompts/work_plan.md",
    "G3_5":      None,
    "G3_6_MANDATES": "_COMMUNICATION/agents_os/prompts/implementation_mandates.md",
    "CURSOR_IMPLEMENTATION": None,  # code commits — not a file check
    "GATE_4":    None,
    "GATE_5":    None,
    "GATE_6":    None,
    "WAITING_GATE6_APPROVAL": None,
    "GATE_7":    None,
    "GATE_8":    "_COMMUNICATION/agents_os/prompts/gate_8_mandates.md",
}
```

**Note on `None` entries:** Gates with `None` are excluded from artifact checks. This is intentional — the check is incremental. As the system matures, more gates get defined artifact paths (Stage B expansion). The check is valuable even for the subset of gates with defined paths.

**Required:** Add `check_gate_integrity()` to `state_reader.py`:

```python

    """
    For each gate in state.gates_completed that has a defined artifact path,
    verify the artifact exists on disk.
    Returns: { "status": "PASS|FAIL", "missing": [...], "checked": [...] }
    """
```

**STATE_SNAPSHOT.json output:**
```json
"consistency_check": {
  "gate_integrity": {
    "status": "PASS",
    "checked_gates": ["GATE_0", "G3_PLAN", "G3_6_MANDATES"],
    "missing_artifacts": [],

  }
```

If `missing_artifacts` is non-empty → `status: "FAIL"`, `severity: "CRITICAL"`.

#### R2.2 — WSM Stage Watch (SOFT WARNING — Nimrod's additional requirement)

**What it detects:** `pipeline_state.stage_id` ≠ WSM `current_stage_id`. This catches teams that edited pipeline state manually or bypassed the stage transition process. This is a **soft warning** (amber), not a hard fail — normal workflow has a window where WSM lags behind. Stage mismatches are rare (S001→S002→S003) and meaningful.

**STATE_SNAPSHOT.json output:**
```json
"consistency_check": {
  "gate_integrity": { ... },
  "wsm_stage_watch": {
    "status": "MATCH",
    "pipeline_stage": "S002",
    "wsm_stage": "S002",
    "severity": "WARNING"
  },
  "overall_status": "PASS"
}
```

**Status logic:**
- `MATCH` → both stages agree → `overall_status` not degraded
- `MISMATCH` → stages differ → `overall_status` = `WARN` (unless gate_integrity already `FAIL`)
- `UNKNOWN` → WSM not found or parse failed → `overall_status` not degraded (no false alarms from missing file)

#### R2.3 — Dashboard Display

**File:** `PIPELINE_ROADMAP.html`

Dashboard reads `STATE_SNAPSHOT.json` and renders banners in the domain header:

| Condition | Banner | Color |
|---|---|---|
| `gate_integrity.status = FAIL` | ⛔ Gate Integrity Violation — artifacts missing. Run: `python3 -m agents_os_v2.observers.state_reader` | Red |
| `wsm_stage_watch.status = MISMATCH` | ⚠️ WSM stage (`{wsm_stage}`) does not match pipeline stage (`{pipeline_stage}`) — verify WSM is current | Amber |
| `overall_status = PASS` | No banner (clean state) | — |

**Acceptance:**
- `build_state_snapshot()` always includes `consistency_check` block
- Gate integrity FAIL surfaces as red banner in dashboard
- WSM mismatch surfaces as amber banner (does NOT block pipeline)
- Both banners auto-clear when condition resolves

### R4 — `pipeline_run.sh new <PROGRAM_ID>` Command
**File:** `pipeline_run.sh` (+ underlying Python)

**Purpose:** Activate a new program in the agents_os domain. Creates a clean pipeline state at GATE_0, ready for the gate chain to begin.

**Command signature:**
```bash
./pipeline_run.sh new <PROGRAM_ID>
# Example:
./pipeline_run.sh new S002-P005
```

**State created:**
```json
{
  "work_package_id": "S002-P005-WP001",
  "project_domain": "agents_os",
  "stage_id": "S002",
  "spec_brief": "",
  "current_gate": "GATE_0",
  "gates_completed": [],
  "gates_failed": []
}
```

**Behavior:**
1. Validate `PROGRAM_ID` format (`S\d{3}-P\d{3}`)
2. Check if a state file already exists for this domain — if so, prompt confirmation (do not overwrite silently)
3. Create `PipelineState(work_package_id=f"{PROGRAM_ID}-WP001", project_domain="agents_os", stage_id=stage, current_gate="GATE_0")`
4. Call `state.save()`
5. Print: `"✅ Program activated: {PROGRAM_ID}-WP001 @ GATE_0"`
6. Print: `"Next: ./pipeline_run.sh --domain agents_os gate GATE_0"`

**Note:** `spec_brief` is set to empty string at activation — populated at GATE_0 when the spec prompt is generated.

**Deferred to Stage B:** `hold`, `cancel`, `revive` subcommands.

**Acceptance:**
- `./pipeline_run.sh new S002-P005` creates `pipeline_state_agents_os.json` with `current_gate: "GATE_0"`
- Running again without state file: creates cleanly
- Running again with existing state file: warns + requires confirmation
