# S003-P002 — TEST TEMPLATE GENERATOR
## LOD200 Architectural Concept

---

## §1 Strategic Purpose

### 1.1 Problem Statement

Every TikTrack feature (from S003 onwards) requires two categories of tests:
1. **API tests** — pytest assertions on backend endpoints (response codes, shapes, edge cases)
2. **E2E tests** — Selenium flows verifying UI behavior (page loads, form interactions, data display)

Currently, Team 50 authors these tests from scratch for each Work Package. This creates:
- **High per-WP test authoring cost** (~30-40% of QA time is scaffolding setup)
- **Inconsistent test structure** across WPs (different fixture patterns, naming, assertion styles)
- **Delayed QA starts** — tests can't begin until Team 50 finishes reading the spec and writing skeleton code

### 1.2 Solution

S003-P002 adds a **Test Template Generator** to agents_os_v2: a code generation module that reads a structured LLD400 spec and emits ready-to-run test scaffold files.

**Not a validator — a generator.** The module takes spec content in and produces Python test files out. Team 50 receives pre-scaffolded files and only needs to fill in assertions, not write boilerplate.

### 1.3 Strategic Fit and Sequencing

Accelerated from S005 because **every S003–S006 TikTrack WP benefits**. The generator amortizes its development cost across all future stages.

```
S003-P002 ships (fast track) → S003 TikTrack WPs (P003..P006) get generated scaffolds → Team 50 QA velocity increases
    ↓
Every S004, S005, S006 WP also gets generated scaffolds
    ↓
By S005: full test infrastructure is template-generated + manually refined = highest signal quality
```

This is the "highest ROI per-token" AGENTS_OS program in S003: a ~1 WP investment pays dividends across 15+ future WPs.

## §2 Architectural Concept

### 2.1 Input: Structured Spec Section

The generator reads a LLD400 spec document and extracts two types of contracts:

**API Contract Block** (parsed from spec tables):
```
| Endpoint | Method | Request | Response | Auth |
| /api/v1/alerts/ | GET | ... | ... | required |
```

**Page/Component Block** (parsed from spec tables):
```
| Page | D34 | Route | /alerts | Components | AlertList, AlertItem |
```

The generator does NOT require any special spec format beyond the existing LLD400 table conventions already used in TikTrack specs. It parses what's already there.

### 2.2 Output: Test Scaffold Files

For each API Contract Block found:
- Generates `tests/api/test_{entity}.py` with fixture + parametrized test stubs

For each Page/Component Block found:
- Generates `tests/ui/test_{page_id}.py` with Selenium setup + flow test stubs

### 2.3 Example: Input → Output

**Input spec (excerpt from D34 LLD400):**
```
| GET /api/v1/alerts/ | Returns paginated alerts list | Auth: JWT required |
| PATCH /api/v1/alerts/{id}/mark-read | Marks alert as read | Auth: JWT + owner check |
```

**Output (auto-generated skeleton):**
```python
# tests/api/test_alerts.py — generated by S003-P002 Test Template Generator
import pytest
from tests.fixtures import auth_client, admin_client

class TestGetAlerts:
    def test_returns_200_authenticated(self, auth_client): ...
    def test_returns_401_unauthenticated(self, client): ...
    def test_response_shape(self, auth_client): ...
    def test_pagination_params(self, auth_client): ...

class TestMarkAlertRead:
    def test_returns_200_owner(self, auth_client): ...
    def test_returns_403_non_owner(self, auth_client_b): ...
    def test_returns_404_unknown_id(self, auth_client): ...
```

Team 50 fills in the assertion bodies. The structure, fixture usage, naming, and test categories are auto-generated.

### 2.4 New Module: `agents_os_v2/generators/`

New directory in agents_os_v2:
```
agents_os_v2/
  generators/
    __init__.py
    test_templates.py     ← primary module
    templates/
      api_test.py.jinja   ← Jinja2 template for API tests
      ui_test.py.jinja    ← Jinja2 template for Selenium tests
```

Domain constraint: generators/ must follow the same domain isolation rules as validators/ — no imports from `api/`.

### 2.5 Pipeline Integration Point

The generator runs as a **GATE_3 sub-step** (new G3.7 sub-stage, after G3.5 work plan validation, before G3.6 team activation mandates):

```
G3.5: Work plan validation (Team 90/190)
G3.7: Test template generation ← NEW (optional step in GATE_3 phase)
        → emits scaffold files to tests/api/ and tests/ui/
        → Team 10 attaches scaffolds to Team 50 activation mandate
G3.6: Team activation mandates (Team 10 issues to Teams 20/30/50)
        → Team 50 receives mandate + scaffold files
```

G3.7 is **optional per WP** — Team 10 can skip if spec has no parseable API/UI contracts. First use: S003-P003 (System Settings) — D39/D40/D41 specs will be the first real generator input.

## §4 Open Architectural Questions (to resolve at LOD400)

These questions are **deferred to LOD400 authoring** (next session after S003-P001 FAST_4):

| # | Question | Options | Impact |
|---|---|---|---|
| Q-01 | Spec parsing: regex vs. structured parser? | (A) Regex on existing table format; (B) Require new structured tag in spec | A = zero friction for spec authors; B = more reliable parsing |
| Q-02 | Template language? | (A) Jinja2; (B) Python string templates; (C) dataclasses-generated | A = most readable templates; B = no new dependency |
| Q-03 | Generated file destination? | (A) Emit to `tests/api/` and `tests/ui/` directly; (B) Emit to `_generated/` staging area for review | A = immediately runnable; B = explicit review step |
| Q-04 | Idempotency? | (A) Overwrite on re-run; (B) Merge (don't overwrite filled assertions) | B requires diff-aware merging — significantly more complex |
| Q-05 | Selenium fixture generation? | (A) Generate full Selenium setup; (B) Only generate test stubs with TODO markers | A assumes Selenium is already installed; B is safer for initial version |

**Team 00 preliminary lean:**
- Q-01: Option A (regex) for initial version — zero spec author friction
- Q-02: Option A (Jinja2) — templates are readable and maintainable
- Q-03: Option A (emit directly) — simplest, Team 50 can git diff to see what changed
- Q-04: Option A (overwrite) for initial version — simpler, Team 50 manages manual portions in separate files
- Q-05: Option B (stubs only) for S003-P002 — full Selenium templates come after validation run

These leans are **not locked** — LOD400 session with Team 100 + Team 61 input finalizes.
