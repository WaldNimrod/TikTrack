## 2. Endpoint Contract

### 2.1 Scope and base URL

- **API base:** `settings.api_v1_prefix` = `/api/v1` (FastAPI mount).
- **Logical resource:** Authenticated user’s watchlist rows in `user_data.user_tickers`, joined to `market_data.tickers`, enriched with live/EOD price fields per existing SSOT (`_get_price_with_fallback` and related).
- **Stakeholder alias:** Any reference to “GET /user_tickers” in legacy language maps to **`GET /api/v1/me/tickers`** (no separate `GET /api/v1/user_tickers` required).

### 2.2 `GET /api/v1/me/tickers` — list (extended)

| Item | Specification |
|------|----------------|
| Method | `GET` |
| Path | `/api/v1/me/tickers` |
| Auth | Required — Bearer JWT / existing session contract. |
| Request body | None. |

**Query parameters (all optional; omission = defaults):**

| Parameter | Type | Default | Constraints |
|-----------|------|---------|-------------|
| `ticker_type` | string, **repeatable** (multi-value) | (none) | Each value ∈ `TICKER_TYPES` SSOT (e.g. `STOCK`, `ETF`, `CRYPTO`, …). OR semantics: row matches if `market_data.tickers.ticker_type` equals any supplied value. |
| `status` | string | (none) | Filters **`user_data.user_tickers.status`**: `active` \| `inactive`. When omitted, no filter on user watchlist status. |
| `sector_id` | string (ULID) | (none) | Filter: `market_data.tickers.sector_id` equals this id. |
| `market_cap_group_id` | string (ULID) | (none) | Filter: `market_data.tickers.market_cap_group_id` equals this id. |
| `sort_by` | string | `display_name` | Allowed: `display_name` \| `company_name` \| `current_price` \| `daily_change_pct` \| `ticker_type` \| `currency` \| `exchange_code` \| `status` \| `price_source` — each MUST correspond to a server-side sortable expression tied to the joined row (see §3.4). |
| `sort_dir` | string | `asc` | `asc` \| `desc`. |
| `page` | integer | `1` | ≥ 1. |
| `page_size` | integer | `25` | ≥ 1, **max 50** (hard cap per D33-IR-03). |

**Sorting nulls (normative):** For `sort_by=display_name` (and other nullable keys), **ASC → nulls last; DESC → nulls first** (per LOD200 risk register).

**Response schema — extended list envelope:**

JSON object:

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `data` | `TickerResponse[]` | Yes | Page slice only (length ≤ `page_size`). |
| `total` | integer | Yes | Total rows matching filters **before** pagination. |
| `page` | integer | Yes | Current page (1-based). |
| `page_size` | integer | Yes | Effective page size (≤ 50). |

**`TickerResponse`** — unchanged field set vs `api/schemas/tickers.py` for this WP (ULID `id`, `symbol`, `display_name`, `company_name`, `ticker_type`, `status`, `is_active`, prices, `price_source`, `currency`, `exchange_code`, etc.). **`symbol` remains in JSON for machine/clients; D33 UI MUST NOT use it as the human primary label** (see §4, Iron Rule).

**Backward compatibility:** Calling `GET /api/v1/me/tickers` **with no query params** MUST behave as: first page, default `page_size` (25), default sort (`display_name` ASC), **no extra filters** — equivalent to prior “full list” semantics, modulo pagination envelope (caller receives `total`, `page`, `page_size` in addition to `data`).

**Errors:** `401` if unauthenticated; `422` on invalid query combination (unknown `sort_by`, `page` &lt; 1, `page_size` &gt; 50, invalid ULID format for filters); `500` / `HTTPExceptionWithCode` per existing patterns.

## 3. DB Contract

### 3.1 Tables (no new DDL)

| Schema.table | Role |
|--------------|------|
| `user_data.user_tickers` | Junction: `user_id`, `ticker_id`, `display_name`, `status`, `notes`, `created_at`, `deleted_at` — **read/write** per operations below. |
| `market_data.tickers` | Ticker metadata — **read** (filters/sort join). |
| `market_data.exchanges` | **Read** for `exchange_code` (and join keys). |
| `market_data.ticker_prices` | **Read** for EOD / last close fields. |
| `market_data.ticker_prices_intraday` | **Read** for intraday/current price where applicable. |
| `market_data.sectors` | **Read** if UI/API resolves sector labels (optional; filter uses `sector_id` on tickers). |
| `market_data.market_cap_groups` | **Read** if UI resolves cap group labels (optional). |

**No new tables, columns, or migrations** for this WP (LOD200 §6.1).

### 3.1.1 Financial precision (NUMERIC(20,8))

**Normative:** Monetary amounts and percentage fields used for D33 live/EOD pricing (including `current_price`, `daily_change_pct`, `last_close_price`, and related columns in `market_data.ticker_prices` and `market_data.ticker_prices_intraday` per the existing SSOT) are stored in PostgreSQL as **`NUMERIC(20,8)`** per Phoenix architectural SSOT; API/ORM layers treat these as **`Decimal(20,8)`**-compatible values. This WP does **not** change column types or add migrations; **read/list responses** MUST preserve this precision and MUST NOT introduce ad-hoc precision or rounding outside the established service-layer rules.

### 3.2 Columns read

| Operation | Tables / columns (minimum) |
|-----------|------------------------------|
| List / page | `user_tickers`: `id`, `user_id`, `ticker_id`, `display_name`, `status`, `deleted_at`; `tickers`: all fields needed for `TickerResponse` + `sector_id`, `market_cap_group_id` for filters; price tables per existing price pipeline. |
| PATCH | `user_tickers` row scoped by `user_id` + `ticker_id`; update `display_name` only (plus audit if any). |
| DELETE | `user_tickers.deleted_at` soft-delete (existing). |
| POST add | Existing insert path into `user_tickers` + ticker resolution. |

### 3.3 Columns written

| Operation | Write |
|-----------|--------|
| GET | None. |
| PATCH | `user_data.user_tickers.display_name` (and only as allowed by service). |
| DELETE | `user_data.user_tickers.deleted_at` (soft delete). |
| POST | New `user_tickers` row (+ ticker create path when symbol-based add). |

### 3.4 Query patterns (normative intent)

- **List:** Join `UserTicker` → `Ticker` (and exchanges) with `user_id = current_user`, `UserTicker.deleted_at IS NULL`, `Ticker.deleted_at IS NULL`.
- **Filters:** Apply `ticker_type`, `user_tickers.status`, `sector_id`, `market_cap_group_id` as AND conditions on the joined row set.
- **Sort:** Single-column sort per `sort_by` / `sort_dir` on resolved SQL expressions (display name for sorting SHOULD use COALESCE consistent with UI rule: prefer `user_tickers.display_name`, then `tickers.company_name` for ordering key — **without** promoting raw `symbol` to the human-facing column; sort key may still use internal columns).
- **Pagination:** `LIMIT/OFFSET` or equivalent: `OFFSET (page - 1) * page_size`, `LIMIT page_size`; `total` from COUNT of filtered set.

### 3.5 Logging

Server-side logging MUST use **`maskedLog`** (or equivalent project SSOT) for user-identifying data — **D33-IR-05**.

## 6. Acceptance Criteria

1. **`GET /api/v1/me/tickers`** returns paginated envelope with `data`, `total`, `page`, `page_size`; `page_size` capped at **50**; default `page=1`, `page_size=25`, default sort `display_name` ASC.  
2. **Filters** `ticker_type` (multi), `status` (user watchlist), `sector_id`, `market_cap_group_id` are supported and combine correctly (AND).  
3. **`sort_by` / `sort_dir`** implement server-side ordering for the enumerated keys; nullable display fields follow **nulls last (ASC)** / **nulls first (DESC)** for `display_name`.  
4. **`PATCH /api/v1/me/tickers/{ticker_id}`** updates `display_name` only (max 100 chars); response is full **`TickerResponse`**.  
5. **D33 primary label** shows `display_name`, else **`company_name`**; raw **`symbol` is never the primary visible identifier** in the table.  
6. **Price column** shows **`current_price`**, **`daily_change_pct`**, and **`price_source`** badge per row.  
7. **Pagination** UI offers **10 / 25 / 50** and never requests more than **50** rows per page.  
8. **Every sortable data column** has a working header sort control (**D33-IR-04**).  
9. **Empty state** and **API error + retry** behaviors are implemented (**D33-IR-06**).  
10. **Add ticker** and **Edit display name** modals complete successfully against **POST** / **PATCH** without inventing non-GIN field names.  
11. **Logging** does not leak raw PII beyond **`maskedLog`** policy (**D33-IR-05**).  
12. **Layout** uses **`tt-container` / `tt-section` / `tt-section-row`** structure (**D33-IR-02**).
