### Scope

**In scope (TikTrack, S003 — D26 Watch Lists, route `/watch_lists`):**

- **Data:** `user_data.watchlists` and `user_data.watchlist_items` only, keyed by ULIDs, scoped to the authenticated user. No cross-user reads or writes.
- **Watchlist CRUD:** Create (name required, description optional), update name/description, delete (confirmation in UI; server deletes list and all items via cascade or explicit delete).
- **Item CRUD:** Add ticker by resolving an existing row in `market_data.tickers` (lookup-only; no ticker creation on this page). Remove item. Reorder via `sort_order` using explicit up/down operations (no drag-and-drop).
- **Limits:** At most **20** watchlists per user; at most **50** tickers per watchlist.
- **Market display:** For the active watchlist, show per row: ticker **display_name** (primary label), current price, day change % (color-coded green/red vs sign), volume, last-updated timestamp for price data, using the same price enrichment approach as D33 (`GET /api/v1/me/tickers` / market-data stack).
- **Client refresh:** Automatic re-fetch of the active list’s price-bearing payload while the page remains mounted (interval locked in Acceptance Criteria).
- **Table UX:** Sort rows by name, price, or % change; client-side text filter over displayed rows (display_name / allowed searchable fields returned by API).
- **Personal only:** `is_public` remains `false` for all rows created in this WP; no UI or API surface exposes public sharing.

**Explicitly excluded:**

- Public or shared watchlists; any use of `is_public = true`.
- ATR(14), position, P/L, P/L% columns (D26-Phase2 / S005).
- Flag-color filter (D26-Phase2 / S005).
- Reordering watchlists themselves (lists selector order fixed by implementation default, e.g. created_at, unless promoted in a future WP).
- Drag-and-drop reordering of tickers.
- Export/import of watchlists.
- Creating tickers not present in `market_data.tickers` (remains D22 / admin flows).
- Agents OS v3 runtime, pipeline UI, and `/api/` (AOS) endpoints — TikTrack product API only under `/api/v1/`.

### Acceptance Criteria

1. **Routing:** An authenticated user with a valid session loads `GET /watch_lists` (or the canonical TikTrack route mapped to D26 in the router config) and receives the watch lists shell with `collapsible-container` and `phoenix-header` per global page template; unauthenticated users receive the same redirect-to-login behavior as other `requiresAuth: true` management pages.
2. **List watchlists:** `GET /api/v1/watchlists` returns **only** watchlists where `user_id` equals the JWT user; response includes each list’s `id` (ULID), `name`, `description` (nullable), and `item_count` (non-negative integer). Response is `200` when authenticated; `401` when not.
3. **Create watchlist:** `POST /api/v1/watchlists` with JSON body `{ "name": "<string>", "description": "<string|null>" }` creates one row; `name` length 1–100 inclusive (matches `VARCHAR(100)`); empty or whitespace-only name yields `422`; success is `201` with the new ULID `id` and persisted fields; creating a 21st watchlist for the same user yields `400` via `HTTPExceptionWithCode` with a new enum member in the TikTrack `ErrorCodes` SSOT dedicated to watchlist count limit (Team 20 registers; name chosen at implementation time without inventing payload field names beyond existing error schema).
4. **Update watchlist:** `PUT /api/v1/watchlists/{id}` with same body constraints as create updates only lists owned by the user; non-owned or unknown `id` yields `404`; success `200`.
5. **Delete watchlist:** `DELETE /api/v1/watchlists/{id}` removes the list and all its items for the owner; non-owned or unknown `id` yields `404`; success `204` or `200` with empty body — one status policy chosen by Team 20 and applied consistently in OpenAPI/tests.
6. **List items with prices:** `GET /api/v1/watchlists/{id}/items` returns ordered items (`sort_order` ascending, tie-break by `added_at` ascending) with each row including `item_id` (ULID), `ticker_id` (ULID), `display_name`, numeric price fields with **at most 8 decimal places** in JSON numbers sourced from DB `NUMERIC(20,8)` where applicable, `daily_change_pct`, `volume`, and `price_as_of` (or the same timestamp field name used on D33 list responses). Unknown or other-user list: `404`. UI renders **no row** using raw `symbol` as the primary label; primary label is `display_name` everywhere in the D26 table.
7. **Add item:** `POST /api/v1/watchlists/{id}/items` with body `{ "ticker_id": "<ULID>" }` adds one row; duplicate `(watchlist_id, ticker_id)` yields `409` or `400` with a dedicated `ErrorCodes` member; ticker not in `market_data.tickers` yields `404`; 51st ticker yields `400` with a dedicated `ErrorCodes` member for item limit; success `201`.
8. **Remove item:** `DELETE /api/v1/watchlists/{id}/items/{item_id}` removes one row for owner; wrong `item_id` or other user: `404`; success `204` or `200` (consistent with §5).
9. **Reorder:** `PUT /api/v1/watchlists/{id}/items/{item_id}/order` with body `{ "direction": "up" | "down" }` swaps `sort_order` with the adjacent row in that direction; at boundary (first/last) the server returns `400` with a dedicated `ErrorCodes` member and leaves order unchanged; success `200` with updated order reflected on next `GET .../items`.
10. **Auto-refresh:** With a watchlist selected and the D26 route mounted, the browser issues a new `GET /api/v1/watchlists/{id}/items` at least once every **120 seconds** (±10 s jitter allowed) until navigation away; each user action that changes items or the selected list triggers an immediate re-fetch of the affected `GET` within 1 s of completion of the mutating request.
11. **Sort controls:** The active list table supports user-triggered sort by display name, price, and daily_change_pct; sort state is deterministic (asc/desc toggle) and reapplied after each data refresh without losing the selected mode.
12. **Filter:** A single text input filters visible rows client-side by substring match (case-insensitive) against `display_name` and the same `symbol` field returned in JSON if present (for search only, not as primary label).
13. **Delete confirmation:** The UI shows an explicit confirmation step before calling delete watchlist; after confirm, the list disappears from the selector and items are gone from DB (verified by subsequent `GET /api/v1/watchlists` and `GET .../items`).
14. **QA evidence:** Team 50 delivers automated or scripted checks covering §2–§9 (contract) and manual or E2E checklist covering §1, §10–§13; zero failing tests in the agreed TikTrack gate suite for this WP before GATE_5 closure.

### Architecture Constraints

- **Identifiers:** All external IDs in API JSON are ULID strings; reject malformed ULIDs with `422`.
- **Money / numeric:** Persist and transmit monetary and fractional quantities using `NUMERIC(20,8)` in the database; API serialization follows existing TikTrack patterns for decimal precision (no new precision looser than 20,8).
- **Display label:** Never present raw ticker symbol as the sole or primary column title for humans; `display_name` is the primary label (D33 Iron Rule, inherited).
- **Page shell:** `collapsible-container` + `phoenix-header`; table rows use established row classes (e.g. `tt-section-row`) per CSS class priority rules.
- **Status model:** Any user-visible status chips on this page use the canonical 4-state adapter (`normalizeToCanonicalStatus` / `toHebrewStatus`) where applicable to ticker or row lifecycle fields exposed by API.
- **Logging:** Server and client use `maskedLog` for sensitive diagnostics (no secrets in clear text).
- **Security:** Every mutating and read endpoint enforces `user_id` = authenticated user; no endpoint accepts a `user_id` from the client for authorization.
- **API prefix:** All routes under `/api/v1/` (not AOS `/api/`).
- **DDL:** No schema change required for S003 scope per Team 110 confirmation recorded in baseline LOD200; if implementation discovers a blocking delta, execution stops and Team 110 issues a delta decision before merge.
