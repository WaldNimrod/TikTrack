## 2) למה הממצאים לא אותרו — ניתוח לפי סוג כשל

### 2.1 כשל 1: אימות API במקום UI (ממצאים 1.1, 1.2, 1.3)

| ממצא | מה בדקנו | מה חסר |
|------|-----------|---------|
| מקור + עודכן ב ריקים | API מחזיר price_source, price_as_of_utc ✓ | **לא וידאנו שהערכים מוצגים בטבלה** — אימות היה על payload, לא על תצוגה |
| רמזור אדום לכולם | logic קוד getTrafficLightFromSource ✓ | **לא הרצנו עם נתונים אמיתיים** — MCP רץ כש-0 טיקרים; כשהתקנו — לא בדקנו ערך הרמזור per row |
| מטבע הכל $ | API מחזיר currency ✓ | **לא וידאנו TEVA.TA→₪, ANAU.MI→€** — הטיקרים בבסיס לא קשורים ל-exchange, נפל ל-USD; לא בדקנו תצוגה מטבעית |

**שורש:** הבדיקות התמקדו ב-**API payload** וב-**קוד**. לא בוצע **Visual Assertion** — אימות שמה שמוצג ב-UI תואם את הציפייה.

---

### 2.2 כשל 2: היעדר בדיקת נתוני אמת (ממצא 1.2, 1.6)

| ממצא | מה בדקנו | מה חסר |
|------|-----------|---------|
| רמזור אדום | הקוד קיים | **נתוני EOD/intraday לא תקינים** — סנכרון/שמירה לא עובד; לא הרצנו בדיקה עם DB מאוכלס ועדכני |
| ניקוי seed | seed script רץ | **לא בדקנו איכות הנתונים** — DDD, TSLA, MSFT מיותרים; SPY/QQQ חסרים; ticker_type שגוי |

**שורש:** הבדיקות הניחו ש**אם הקוד קיים — זה עובד**. לא היתה בדיקת **Data Fidelity** — וידוא שנתוני הבסיס תואמים את הדרישות.

---

### 2.3 כשל 3: Scope צר מדי (ממצאים 1.4, 1.5, 1.7, 1.8)

| ממצא | מה בדקנו | מה חסר |
|------|-----------|---------|
| מודל פרטים | handleDetails קיים, data-integrity 200 | **לא בדקנו תוכן המודל** — אילו שדות מוצגים; דרישה: "כל שדות עריכה + נתוני שוק מלאים" |
| סוג נכס STOCK | API מחזיר ticker_type | **לא בדקנו דוגמאות ETF** — SPY, QQQ חסרים; כל הקיימים STOCK |
| הוספה — מטבע + בורסה | טופס קיים | **לא בדקנו טופס הוספה מול אפיונים** — symbol+exchange (ANAU.MI = ANAU + .MI) |
| OPTION/FUTURE | לא בבדיקות | **לא תועד** — המערכת אינה תומכת; UI/API עשויים להראות כאילו נתמך |

**שורש:** ה-scope היה **BF-001..004 בלבד**. לא הורחב ל:
- Full UX per spec (כל השדות, כל הפעולות)
- Data quality (ticker_type, exchange, currency)
- Documentation of limitations (OPTION/FUTURE)

### 2.4 כשל 4: MCP ללא נתונים + סשן שונה (ממצאים 1.1–1.3)

**שורש:** **תלות בסביבה** — הבדיקות לא רצו בסביבה מזוהמת (seeded, admin auth) שתתאים למה ש-Nimrod רואה.

## 3) המלצות לשיפור — סוויטת בדיקות אוטומטית (GATE_4 + GATE_5)

### 3.1 עקרון: מקסימום אוטומציה, UI כשורה תחתונה

- **כל השכבות:** API, DB, Frontend logic — אוטומטי.
- **שורה תחתונה:** ממשק המשתמש הסופי — חובה לכסות באוטומציה (MCP, Playwright, או E2E דומה).
- **GATE_4 ו-GATE_5:** מריצים את **אותה סוויטה** (או GATE_5 = סופרסט של GATE_4). ללא לופ ידני.

### 3.2 סוויטת בדיקות UI אוטומטית — D22 Tickers (דוגמה)

להפוך את ה-Checklist לסקריפט/תרחישים אוטומטיים:

| # | בדיקה | שכבה | איך לאוטומט |
|---|--------|------|--------------|
| 1 | Login as Admin | E2E | MCP / Playwright — fill + submit |
| 2 | Seed נתונים (TEVA.TA, BTC-USD, ANAU.MI, SPY, AAPL) | DB | `make seed-tickers` לפני הרצה |
| 3 | עמודת "מקור" לא ריקה | UI | Snapshot / DOM query — assert text content |
| 4 | עמודת "עודכן ב" לא ריקה | UI | Snapshot / DOM query — assert |
| 5 | רמזור — לא אדום לכולם | UI | Assert לפחות שורה אחת עם class ירוק/צהוב |
| 6 | מטבע — TEVA.TA→₪, ANAU.MI→€, BTC-USD→$ | UI | Assert סימן מטבע בתא |
| 7 | Details modal — שדות עריכה + נתוני שוק | UI | Click "פרטים" → assert DOM מכיל שדות |
| 8 | טופס הוספה — שדות מטבע, בורסה | UI | Assert presence של אלמנטים |
| 9 | ticker_type נכון (ETF ל-SPY/QQQ) | API/DB | Assert ב-payload או בטבלה |

### 3.3 Data Fidelity — כחלק מהאוטומציה

- **Pre-step:** הרצת seed קנוני לפני כל run.
- **Assertions:** ticker_type, exchange_id, currency ב-DB או ב-API.
- **Price source:** אחרי sync — assert price_source לא null ל־≥1 טיקר.

### 3.4 Nimrod — תפקיד נקודתי (לא לופ)

- **בשגרה:** GATE_5 עובר/נכשל **רק** לפי תוצאות האוטומציה.
- **בקשה נקודתית:** ניתן לבקש מ-Nimrod — "הרץ X" או "בדוק נקודה Y" — כשנדרש, ללא תלות בשער.
- **GATE_7:** אישור אנושי סופי — צפוי להיות מהיר כי האוטומציה כיסתה את הרוב.
