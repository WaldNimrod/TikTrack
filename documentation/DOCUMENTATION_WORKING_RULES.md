# כללי עבודה עם מערכת הדוקומנטציה - TikTrack

## 🎯 כלל מרכזי

**כל הדוקומנטציה שלנו נמצאת תחת תקיית `documentation/` בלבד - אין ליצור קבצי דוקומנטציה מחוץ לה**

> **עקרון בסיסי:** כל עבודה על דוקומנטציה חייבת להתבצע מול המערכת המסודרת תחת `documentation/` בלבד

## 📋 כללים בסיסיים

### ✅ מה לעשות

1. **עבודה מול תקיית documentation** - כל הקבצים תחת `documentation/`
2. **עדכון אינדקסים** - כל קובץ חדש חייב להיות מקושר ב-`INDEX.md`
3. **שמות קבצים ברורים** - שם הקובץ חייב לשקף את התוכן
4. **מיקום נכון** - קובץ חייב להיות בתיקייה המתאימה לו

### ❌ מה לא לעשות

1. **יצירת קבצים מחוץ ל-documentation** - אין ליצור קבצי MD מחוץ לתקייה
2. **כפילויות** - אין ליצור קבצים עם תוכן זהה
3. **קבצים ללא קישור** - אין ליצור קבצים שלא מקושרים באינדקס
4. **שמות לא ברורים** - אין ליצור קבצים עם שמות לא מובנים

## 🔗 מבנה תקיות

```
documentation/
├── INDEX.md                    # אינדקס ראשי
├── 02-ARCHITECTURE/           # ארכיטקטורה
├── 03-DEVELOPMENT/            # פיתוח
├── 04-FEATURES/               # תכונות
├── 05-REPORTS/                # דוחות
├── frontend/                  # Frontend
├── backend/                   # Backend
└── pages/                     # עמודים ספציפיים
```

## 📝 תהליך עבודה

1. **בדיקה** - האם הקובץ כבר קיים?
2. **מיקום** - איפה הקובץ צריך להיות?
3. **יצירה/עדכון** - יצירת הקובץ או עדכון קיים
4. **קישור** - הוספת קישור לאינדקס המתאים
5. **בדיקה** - וידוא שהקישור עובד

## 📝 Markdown Quality Standards

### כללי איכות Markdown

כל קבצי ה-markdown בפרויקט חייבים לעבור בדיקת markdownlint לפני commit ו-push.

#### בדיקה לפני Commit

**חובה להריץ לפני כל commit של קבצי markdown:**

```bash
npm run markdownlint:check
```

#### תיקון אוטומטי

**רוב השגיאות ניתנות לתיקון אוטומטי:**

```bash
npm run markdownlint:fix
```

#### כללי כתיבה

1. **אורך שורה**: מקסימום 120 תווים (לא חל על code blocks)
2. **סגנון כותרות**: ATX style (`#` במקום `===`)
3. **הזחה**: 2 רווחים (לא tabs)
4. **רווחים**: רווחים סביב כותרות, רשימות ו-code blocks
5. **Code blocks**: חייבים לכלול שפת תכנות (למשל ` ```python`)

#### תיקיות מוחרגות

התיקיות הבאות מוחרגות מבדיקות markdownlint:

- `documentation/05-REPORTS/` - דוחות אוטומטיים עם פורמטים שונים
- `documentation/audits/` - קבצים אוטומטיים
- `.cursor/` - תצורת IDE
- `node_modules/`, `.git/`, `archive/`, `backup/` - תיקיות מערכת

#### קבצים שנבדקים

markdownlint בודק את כל קבצי ה-markdown בפרויקט:

- כל קבצי `documentation/` (חוץ מהתיקיות המוחרגות)
- README files (בשורש, tests/, וכו')
- קבצי מדריכים (`trading-ui/*.md`, `tests/*.md`)
- כל קבצי `.md` אחרים בפרויקט

#### Bypass (רק במקרי חירום)

```bash
SKIP_MARKDOWNLINT=1 git push
```

## 🚨 זכור

- **הקוד הוא מקור האמת** - דוקומנטציה חייבת לשקף את הקוד בפועל
- **עדכניות** - דוקומנטציה מיושנת יותר גרועה מדוקומנטציה חסרה
- **פשטות** - דוקומנטציה ברורה וקצרה עדיפה על מפורטת ומבלבלת
- **איכות** - כל קבצי markdown חייבים לעבור markdownlint לפני commit
