## 1. מטרה

מסמך זה מגדיר את תהליך הבדיקות התלת-שכבתי המחייב בפרויקט, במטרה להבטיח איכות מקסימלית בכל שלב.

---

## 1א. תנאי חריגה: סדר ביצוע לאחר כל פיתוח חדש

**חובה לאחר כל השלמת תוכנית או שלב פיתוח:**

1. **צוות 10 (השער)** — מאשר השלמה ומבצע בדיקות ראשוניות (וידוא דיווחי צוותים, עקביות לתוכנית, אין חסימות מדווחות).
2. **רק לאחר מכן** — **צוות 50 (QA)** מבצע בדיקות (שער א'). אין לסגור שלב או להפעיל שלב הבא בלי שער א'.

**כלל ברזל:** אין סגירת שלב/תוכנית בלי QA צוות 50; ו-QA צוות 50 מתבצע **אחרי** שאישר צוות 10 והבדיקות הראשוניות בוצעו.

---

## 1ב. מסירת קונטקסט מצוות 10 לצוות 50 (חובה)

**צוות 50 אינו חלק מלופ סבב הפיתוח.** כדי שצוות 50 יבצע QA — צוות 10 **חייב** למסור לו עדכון מפורט הכולל קונטקסט מלא:

| פריט | חובה |
|------|--------|
| **מה פותח** | תיאור ברור של התוכנית/השלב ומה הושלם (תוכנית עבודה, משימות, צוותים). |
| **מה נדרש לבדוק** | ה scope של הבדיקות: אילו עמודים/API/תזרים, אילו Gates רלוונטיים (A/B/C). |
| **קונטקסט** | קישורים ל-SSOT רלוונטי (תוכנית, ADR, Page Tracker), Endpoints/פורטים, והנחיות רלוונטיות. |
| **מיקום** | הודעה ב-`_COMMUNICATION/team_10/` המכוונת ל-Team 50 (או `team_50/`) עם הקונטקסט המלא. |

**ללא עדכון מפורט זה — צוות 50 לא מניח שהוא "בקיא" בסבב הפיתוח האחרון ולא יתחיל QA ללא קבלת הקונטקסט מצוות 10.**

## 2. שערי האיכות (Quality Gates)

### ✅ שער א': בדיקות אוטומטיות (צוות 50) — סבב ראשון

| פריט | תיאור |
|------|--------|
| **אחריות** | צוות 50 (QA & Fidelity). |
| **מהות** | הרצת כל סוויטת הבדיקות האוטומטיות המוגדרת ב-`TEAM_50_QA_WORKFLOW_PROTOCOL.md`. |
| **כולל** | בדיקות אינטגרציה, Code Review, בדיקות Runtime (Selenium), ובדיקות E2E. |
| **סבב ראשון — תיקונים ישירים:** צוות 50 **מחזיר תיקונים ישירות לצוותים** (20, 30, 60 וכו') לפי הממצאים; הצוותים מתקנים ומדווחים. אין סגירת שער א' לפני שכל התיקונים טופלו ומעבר הבדיקות הושלם. ראה גם `TT2_TEAM_50_DEFECT_REPORTING_PROCEDURE`. |
| **תוצר** | דוח סיכום המאשר מעבר חלק של כל הבדיקות האוטומטיות (0 SEVERE); או רשימת תיקונים פתוחה עד להשלמה. |
| **סטטוס מעבר** | `GATE_A_PASSED` |

### ✅ שער ב': ולידציה — ביקורת חיצונית (צוות 90, המרגל)

| פריט | תיאור |
|------|--------|
| **אחריות** | צוות 90 (The Spy). |
| **מהות** | **אחרי** ששער א' (צוות 50) הושלם — מבוצעת **ולידציה** ביקורת חיצונית בסביבת codex: חוסן, אבטחה, עמידה בסטנדרטים ארכיטקטוניים. |
| **סדר:** | סבב ראשון = צוות 50 (תיקונים ישירים לצוותים) → **אחר כך** ולידציה ע"י המרגל (צוות 90). |
| **תוצר** | דוח ביקורת המאשר עמידה בכל הדרישות. |
| **סטטוס מעבר** | `GATE_B_PASSED` |

### ✅ שער ג': אישור ויזואלי סופי (G-Lead)

| פריט | תיאור |
|------|--------|
| **אחריות** | G-Lead (נמרוד) — **היחיד** שמבצע אישור ויזואלי בפועל. האדריכלית אינה רואה את הממשק בעין אנושית. |
| **מהות** | לאחר מעבר שני השערים הראשונים, מבוצעת בדיקה ידנית-ויזואלית סופית של המוצר בדפדפן. |
| **כולל** | השוואה ויזואלית ל-Blueprint, בדיקת חווית משתמש (UX), ואישור סופי של המראה והתחושה. |
| **תוצר** | אישור סופי (Sign-off) או החזרת רשימת תיקונים ויזואליים. |
| **סטטוס מעבר** | `FINAL_APPROVAL` |
| **דיוקי עיצוב / תיקוני עיצוב** | **דיוקי עיצוב ותיקוני עיצוב (תפריט, מודולים, וכל מקום)** — הצוותים **חייבים** לבצע **מול G-Lead** בלבד. לא מואצלים לאדריכלית; האישור הויזואלי בפועל מבצע G-Lead. |
| **אמצעי אימות** | **אין שימוש בצילומי מסך** לאישור ויזואלי — האישור מתבצע בדפדפן מול G-Lead. |
