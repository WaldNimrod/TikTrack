## 2) Pipeline Usage Guide

The V2 Pipeline CLI is the deterministic replacement for manual Team 10 chat. Run from repo root.

### 2.1 Entry point

```bash
python3 -m agents_os_v2.orchestrator.pipeline <command> [options]
```

### 2.2 Commands

| Command | Purpose |
|---------|---------|
| `--spec "feature description"` | Start a new pipeline run with a short spec; initializes state and sets the work scope. |
| `--status` | Show current pipeline state (spec, stage, WP, current gate, last event). |
| `--next` | Output the next recommended action (e.g. which gate to run, which team to activate). |
| `--generate-prompt GATE_X` | Generate the full 4-layer prompt for the given gate; writes to `_COMMUNICATION/agents_os/prompts/` and prints path. |
| `--advance GATE_X PASS \| FAIL` | Record gate outcome and advance state (or mark failure and hold). |

### 2.3 Typical flow

**Canonical gate model:** `_COMMUNICATION/_Architects_Decisions/ARCHITECT_DIRECTIVE_GATE_SEQUENCE_CANON_v1.0.0.md` (5 gates: GATE_1..GATE_5).

1. **Start:** `python3 -m agents_os_v2.orchestrator.pipeline --spec "Add feature X"`
2. **Check:** `--status` and `--next` to see current gate and next step.
3. **Prompt:** `--generate-prompt GATE_3` (or GATE_4, GATE_5, etc.) — use the generated prompt in Cursor/Codex/Claude as specified. Legacy IDs (G3_PLAN, GATE_8) map per migration table.
4. **After gate completes:** `--advance GATE_4 PASS` (or FAIL).
5. Repeat until GATE_5 documentation closure.

## 4) Context injection (4 layers)

Every prompt generated by V2 includes **four layers** so that agents do not need to build context themselves:

| Layer | Content |
|-------|--------|
| **Identity** | Team identity (from `agents_os_v2/context/identity/team_*.md`): role, scope, authority. |
| **Governance** | Gate rules, conventions, and contracts (from `agents_os_v2/context/governance/`, conventions). |
| **State** | Current pipeline state: spec, stage, WP, current gate, last event (from pipeline state file). |
| **Task** | Concrete task for the current step (e.g. validate work plan, run QA scenarios, approve intent). |

**Rule:** When working from a V2-generated prompt, use it as-is. Do not strip layers or rebuild context manually unless the pipeline is not in use for that run.

## 7) WSM Recovery Procedure (G-08 — DM-005 hardening)

The WSM (`PHOENIX_MASTER_WSM_v1.0.0.md`) is written exclusively by `write_wsm_state()` and
`write_wsm_idle_reset()`.  External tools (e.g. Cursor Composer sessions with stale context)
may overwrite it with ghost WP values.  When `ssot_check` reports drift, use this sequence:

### 7.1 Standard recovery (5 seconds)

```bash
# Step 1 — restore committed WSM version
git checkout HEAD -- documentation/docs-governance/01-FOUNDATIONS/PHOENIX_MASTER_WSM_v1.0.0.md

# Step 2 — write clean idle state over it
./pipeline_run.sh wsm-reset           # use --domain agents_os or --domain tiktrack if needed

# Step 3 — verify
python3 -m agents_os_v2.tools.ssot_check --domain agents_os
python3 -m agents_os_v2.tools.ssot_check --domain tiktrack
# Expected: SSOT CHECK: ✓ CONSISTENT on both
```

### 7.2 When to use

| Symptom | Action |
|---------|--------|
| `ssot_check` shows `WSM='S003-P011-WP099'` | Run standard recovery above |
| `current_gate: WSM='GATE_3' ≠ pipeline='COMPLETE'` | Run standard recovery above |
| `in_progress_work_package_id` or `active_program_id` non-N/A when COMPLETE | Run `./pipeline_run.sh wsm-reset` |
| `git diff` shows WSM changed but you didn't touch it | External editor reverted it — run step 1+2 |

### 7.3 Prevention

- **Never edit WSM by hand.** Always use `wsm-reset` or `pipeline_run.sh pass/fail`.
- **COMPLETE guard:** `pipeline_run.sh pass` and `fail` are blocked when gate=COMPLETE — safe by design.
- **Close stale Cursor sessions** that have an old WSM version cached — they revert the file when touched.
- **`run_canary_safe.sh`** is the canonical safe check: runs `generate_mocks + verify_layer1 + ssot_check` with **no WSM writes**.

## 8) GATE_2 Phase Reference (G-09 — operator alignment)

GATE_2 is a five-phase architectural review gate. The engine and all operator documents use
this exact phase sequence:

| Phase | Owner | Purpose |
|-------|-------|---------|
| **2.1** | Team 101 / Team 100 | LOD200 architectural review — is the spec buildable? |
| **2.1v** | Team 190 | Constitutional validation of 2.1 review |
| **2.2** | Team 11 / Team 10 | Work plan production (mandate generation) |
| **2.2v** | Team 90 | Work plan review and validation |
| **2.3** | Team 101 / Team 100 | Final architectural sign-off → GATE_3 |

### 8.1 Precision pass syntax (required for GATE_2)

```bash
# Phase 2.1 pass
./pipeline_run.sh --domain agents_os --wp <WP_ID> --gate GATE_2 --phase 2.1 pass

# Phase 2.1v pass
./pipeline_run.sh --domain agents_os --wp <WP_ID> --gate GATE_2 --phase 2.1v pass

# Phase 2.2 pass  (after Team 11 stores work plan)
./pipeline_run.sh --domain agents_os --wp <WP_ID> --gate GATE_2 --phase 2.2 pass

# Phase 2.2v pass
./pipeline_run.sh --domain agents_os --wp <WP_ID> --gate GATE_2 --phase 2.2v pass

# Phase 2.3 pass  → advances to GATE_3
./pipeline_run.sh --domain agents_os --wp <WP_ID> --gate GATE_2 --phase 2.3 pass
```

> **Note:** `--wp`, `--gate`, `--phase` are precision safety flags (KB-84).
> If the active state does not match, the pass is blocked with a clear error message.

### 8.2 Engine entry vs. display phases

The pipeline engine's `_GATE_ENTRY_PHASE` for GATE_2 is `2.2` (the work-plan production phase).
Phases 2.1 and 2.1v are the architectural review sub-phases that precede it.
All five phases (2.1 → 2.1v → 2.2 → 2.2v → 2.3) must complete before GATE_3 unlocks.
