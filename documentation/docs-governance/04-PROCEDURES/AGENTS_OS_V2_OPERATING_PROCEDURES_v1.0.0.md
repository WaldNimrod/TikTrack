## 2) Pipeline Usage Guide

The V2 Pipeline CLI is the deterministic replacement for manual Team 10 chat. Run from repo root.

### 2.1 Entry point

```bash
python3 -m agents_os_v2.orchestrator.pipeline <command> [options]
```

### 2.2 Commands

| Command | Purpose |
|---------|---------|
| `--spec "feature description"` | Start a new pipeline run with a short spec; initializes state and sets the work scope. |
| `--status` | Show current pipeline state (spec, stage, WP, current gate, last event). |
| `--next` | Output the next recommended action (e.g. which gate to run, which team to activate). |
| `--generate-prompt GATE_X` | Generate the full 4-layer prompt for the given gate; writes to `agents_os_output/prompts/` and prints path. |
| `--advance GATE_X PASS \| FAIL` | Record gate outcome and advance state (or mark failure and hold). |

### 2.3 Typical flow

1. **Start:** `python3 -m agents_os_v2.orchestrator.pipeline --spec "Add feature X"`
2. **Check:** `--status` and `--next` to see current gate and next step.
3. **Prompt:** `--generate-prompt G3_PLAN` (or GATE_4, GATE_5, etc.) — use the generated prompt in Cursor/Codex/Claude as specified.
4. **After gate completes:** `--advance GATE_4 PASS` (or FAIL).
5. Repeat until GATE_8 documentation closure.

## 4) Context injection (4 layers)

Every prompt generated by V2 includes **four layers** so that agents do not need to build context themselves:

| Layer | Content |
|-------|--------|
| **Identity** | Team identity (from `agents_os_v2/context/identity/team_*.md`): role, scope, authority. |
| **Governance** | Gate rules, conventions, and contracts (from `agents_os_v2/context/governance/`, conventions). |
| **State** | Current pipeline state: spec, stage, WP, current gate, last event (from pipeline state file). |
| **Task** | Concrete task for the current step (e.g. validate work plan, run QA scenarios, approve intent). |

**Rule:** When working from a V2-generated prompt, use it as-is. Do not strip layers or rebuild context manually unless the pipeline is not in use for that run.

## 5) MCP integration

### 5.1 Role in development

- **G3_6 / CURSOR_IMPLEMENTATION:** Cursor Composer sessions (Teams 20/30) implement; MCP browser tests can be run during implementation to verify UI flows.
- **GATE_4:** Team 51 executes AGENTS_OS QA including MCP browser test scenarios (18 entities in `agents_os_v2/mcp/test_scenarios.py`). Evidence validator (`agents_os_v2/mcp/evidence_validator.py`) can be used to validate materialization evidence.

### 5.2 How MCP fits the gate flow

1. **Implementation phase:** Developers use Cursor with MCP; scenarios are available for ad-hoc or scripted runs.
2. **GATE_4:** Team 10 hands off to Team 51 with test scenarios for AGENTS_OS lane; Team 51 runs E2E/MCP scenarios as per handover; results feed into GATE_5 evidence. (For SHARED/TIKTRACK lane, Team 50 remains QA owner.)
3. **Evidence:** MATERIALIZATION_EVIDENCE and related artifacts can be validated by the V2 evidence validator before GATE_5/GATE_6 submission.

### 5.3 Canonical scenario set

- Defined in `agents_os_v2/mcp/test_scenarios.py` (18 entities). Do not change the entity set without Team 61/Team 00 alignment; extend only via governance change.
