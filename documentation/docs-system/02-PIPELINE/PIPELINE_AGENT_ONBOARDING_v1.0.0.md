## §1 — What You Need to Know

- You are **one team** in a **multi-team pipeline**. Other teams produce inputs; you produce **one** deliverable per mandate.
- Your **mandate** (markdown) is generated into `_COMMUNICATION/agents_os/prompts/…` — read the section with **your** team number.
- **⛔ IRON RULE:** You **never** run `./pipeline_run.sh` or any pipeline CLI. Only the operator (Nimrod) advances gates (`pipeline.py:64-81` HITL block).
- **Output path** is always under **“Output — write to:”** or `writes_to` in `MandateStep` (`pipeline.py:527-567`).
- Save your artifact → **notify Nimrod** → stop.

## §2 — Canonical File Naming

**Pattern:** `TEAM_{NN}_{WP_SNAKE}_{PURPOSE}_v{VERSION}.md`

**Examples (valid):**

- `TEAM_170_S003_P013_WP001_LLD400_v1.0.0.md`
- `TEAM_190_S003_P013_WP001_GATE_0_VALIDATION_v1.0.0.md`

**Anti-patterns:** Missing `TEAM_` prefix; wrong WP; versionless filename; saving to `documentation/` when mandate says `_COMMUNICATION/team_NN/`.

**Derive WP** from mandate identity header (`gate`, `wp`, `stage`).

## §3 — Verdict Format Standards

Dashboard detection is centralized in `extractVerdictStatus()` (`agents_os/ui/js/pipeline-dashboard.js:1909-1946`). Put the decisive phrase **in the first ~600 characters** when using JSON (KB-76: JSON block only scanned at file start).

### §3.1 — Validation teams (Team 190, Team 90)

Use **one** of these (recognized as PASS):

- `**status:** PASS` or `**status:** APPROVED` (YAML-style header — `pipeline-dashboard.js:1920-1924`)
- JSON at top of file: `{"decision":"PASS"}` or `"decision":"APPROVED"` (first 600 chars only — `pipeline-dashboard.js:1911-1918`)
- Phrase: `VERDICT: PASS`, `FINAL DECISION: PASS`, or `DECISION: PASS` (step 4 — `pipeline-dashboard.js:1939`)

Use **one** of these (recognized as BLOCK):

- `**status:** BLOCK` or `**status:** REJECTED`
- JSON: `"decision":"BLOCK"` or `"decision":"REJECTED"`
- `VERDICT: BLOCK` / `FINAL DECISION: BLOCK`
- `## Overall Verdict` section with `PASS` or `BLOCK` in the following excerpt (`pipeline-dashboard.js:1926-1931`)
- `BLOCKING_REPORT` (word match → BLOCK — `pipeline-dashboard.js:1944`)
- At least one `BF-…` blocking finding (inferred BLOCK — `pipeline-dashboard.js:1933-1936`)

### §3.2 — Documentation teams (Team 70, Team 170)

Closure / AS_MADE style:

- **`CLOSURE_RESPONSE — PASS`** or **`CLOSURE_RESPONSE: PASS`** (em dash or colon — `pipeline-dashboard.js:1941-1942`)
- **`CLOSURE_RESPONSE — BLOCK`** for block
- For constitutional validators, **Team 170** may also use the same **`VERDICT:` / `**status:**`** patterns as §3.1 when the mandate expects a formal verdict file.

### §3.3 — QA teams (Team 50, Team 51)

- Prefer explicit **`**status:** PASS`** or **`**status:** BLOCK`** at top of QA report.
- Use **`BLOCKING_REPORT`** in the filename or body when issuing a formal block package (matches Team 90 pattern `TEAM_90_*_BLOCKING_REPORT*.md` — `pipeline-dashboard.js:1347` candidate list).
- List **`BF-`**-style findings for blockers so the dashboard can infer BLOCK if the header is missing (`pipeline-dashboard.js:1935-1936`).

### §3.4 — Mandate-required phrases (checklist)

| Pattern | Role |
|--------|------|
| `VERDICT: PASS` | Explicit PASS (step 4 — `pipeline-dashboard.js:1939`) |
| `CLOSURE_RESPONSE — PASS` | Doc closure PASS (step 5 — `pipeline-dashboard.js:1941-1942`) |
| `BLOCKING_REPORT` | BLOCK signal (step 5 — `pipeline-dashboard.js:1944`) |
| `FINAL DECISION: APPROVED` | Use **`{"decision":"APPROVED"}`** or **`**status:** APPROVED`** — both map to PASS (steps 0–1 — `pipeline-dashboard.js:1914-1924`). Step 4 only matches `FINAL DECISION` + **`PASS`** or **`BLOCK`**, not `APPROVED`. |
| `{"decision":"PASS"}` | JSON PASS (first 600 chars — `pipeline-dashboard.js:1911-1918`) |

### §3.5 — `route_recommendation` (when BLOCK)

On **BLOCK**, include a short **routing recommendation** so the operator can run `./pipeline_run.sh route doc GATE_N` vs `route full GATE_N` — **must** align with `ALL_GATE_DEFS[gate].failRoutes.doc` / `.full` (`agents_os/ui/js/pipeline-dashboard.js:936+` for `failRoutes`; commands are generated by `pipeline_run.sh` `route` subcommand). Example:

```text
route_recommendation: doc
```

Use **`doc`** for documentation-only remediation; **`full`** when the work package must be reworked or re-planned at a deeper gate (exact labels per gate are in the dashboard gate definitions).
