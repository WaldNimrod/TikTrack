## 1) Purpose

Define deterministic Team 90 checkpoints for validating `MATERIALIZATION_EVIDENCE.json` in `S002-P002` for:

1. **GATE_5 DEV_VALIDATION entry/decision support**
2. **GATE_6 execution submission support**

This protocol locks evidence quality and traceability before gate decisions.

---

## 2) Scope and boundary

- **In scope:** evidence structure, signature/provenance integrity, gate context linkage, parity/result traceability, artifact path integrity.
- **Out of scope:** replacing human authority in GATE_7 or Team 90 closure authority in GATE_8.
- **Boundary lock:** MCP evidence is **advisory only** for GATE_7; Team 90 remains gate owner for GATE_5-8.

---

## 3) Required input package for Team 90 validation

Each submission must include:

1. `MATERIALIZATION_EVIDENCE.json` (signed).
2. Referenced runtime artifacts/logs/results (by path).
3. Gate context artifact (request/handoff package).
4. Relevant checkpoint sign-off artifacts (G3.5/G3.6 and later gate artifacts).

Missing any item => protocol result `BLOCK`.

---

## 4) Checkpoints (deterministic)

### 4.1 Contract and integrity checkpoints (EVC)

| ID | Check | PASS condition | Failure class |
|---|---|---|---|
| EVC-01 | Required top-level keys | Evidence includes provenance, signature block, gate context, artifact path set | BLOCK |
| EVC-02 | Provenance tag | `provenance` in {`TARGET_RUNTIME`,`LOCAL_DEV_NON_AUTHORITATIVE`,`SIMULATION`} (per Evidence Contract) | BLOCK |
| EVC-03 | Signature presence | Signature block contains `key_id`, `signature_base64`, `signed_payload_sha256`, `signed_at_utc`, `signed_by_team` | BLOCK |
| EVC-04 | Signature consistency | Signed payload hash matches materialized payload hash | BLOCK |
| EVC-05 | Timestamp validity | `signed_at_utc` is valid ISO8601 and not empty | BLOCK |
| EVC-06 | Team attribution | `signed_by_team` matches emitting team in handoff package | BLOCK |
| EVC-07 | Artifact path resolvability | All referenced artifact paths exist in repo or declared runtime path list | BLOCK |

### 4.2 Gate context checkpoints (GVC)

| ID | Check | PASS condition | Failure class |
|---|---|---|---|
| GVC-01 | Program match | `program_id` is `S002-P002` | BLOCK |
| GVC-02 | Gate alignment | Evidence declares target gate context (`GATE_5` or `GATE_6`) correctly | BLOCK |
| GVC-03 | Upstream checkpoint linkage | Evidence references mandatory prior checkpoints/sign-offs | BLOCK |
| GVC-04 | Decision traceability | Each evidence item maps to a concrete expected gate criterion | BLOCK |

### 4.3 Result quality checkpoints (RQC)

| ID | Check | PASS condition | Failure class |
|---|---|---|---|
| RQC-01 | Parity/result declaration | Evidence explicitly states parity/result outcome with numeric summary where applicable | BLOCK |
| RQC-02 | Reproducibility hint | Evidence records runtime identity/context (`RUNTIME_IDENTITY` or equivalent) | BLOCK |
| RQC-03 | Contradiction scan | No internal contradiction between declared status and attached artifacts | BLOCK |
| RQC-04 | Advisory tag for MCP | MCP-origin outputs are marked advisory and not represented as human sign-off | BLOCK |

## 5) Gate-specific acceptance rules

### 5.1 GATE_5

GATE_5 evidence is acceptable only if:

1. All `EVC-*`, `GVC-*`, `RQC-*` pass.
2. Evidence supports DEV validation scope for active WP/package.
3. No unresolved contradiction between QA outputs and materialization evidence.

Outcome model for Team 90:
- `PASS` — evidence accepted for GATE_5 decision package.
- `BLOCK` — missing/invalid evidence; remediation required.

### 5.2 GATE_6

GATE_6 evidence is acceptable only if:

1. All `EVC-*`, `GVC-*`, `RQC-*` pass.
2. Evidence-by-path is stable and submission-ready for architect routing.
3. Any advisory-only evidence is clearly non-authoritative for human sign-off.

Outcome model for Team 90:
- `PASS` — evidence accepted for GATE_6 routing package.
- `BLOCK` — package incomplete or non-deterministic.
